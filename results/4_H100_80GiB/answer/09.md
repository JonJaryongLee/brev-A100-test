# Prompt
PostgreSQL 데이터베이스 성능 최적화 방법을 알려줘.

# Answer
PostgreSQL 데이터베이스의 성능을 최적화하는 것은 애플리케이션의 응답 속도, 처리량, 자원 사용 효율성을 높이는 데 매우 중요합니다. 아래는 PostgreSQL 성능 최적화를 위한 주요 방법들을 정리한 것입니다:

---

## 🔧 1. **인덱스(Index) 최적화**

### ✅ 인덱스 생성 전략
- 자주 사용되는 `WHERE`, `JOIN`, `ORDER BY` 절에 있는 컬럼에 인덱스를 생성하세요.
- 예: `SELECT * FROM users WHERE email = 'user@example.com';`

```sql
CREATE INDEX idx_users_email ON users(email);
```

### 🚫 인덱스 과다 사용 피하기
- 인덱스는 읽기 성능을 향상시키지만, 쓰기(INSERT/UPDATE/DELETE) 성능은 저하시킬 수 있습니다.

### 🔄 복합 인덱스 (Composite Index)
- 여러 컬럼을 함께 사용하는 쿼리에 유용합니다.

```sql
CREATE INDEX idx_orders_customer_date ON orders(customer_id, order_date);
```

---

## 📊 2. **쿼리 최적화**

### ✅ EXPLAIN ANALYZE 사용
- 쿼리 실행 계획을 확인하여 성능 문제를 파악하세요.

```sql
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'user@example.com';
```

### 🚫 N+1 쿼리 문제 방지
- JOIN이나 서브쿼리를 적절히 사용하여 N+1 문제를 피하세요.

### 🧠 쿼리 캐싱
- 자주 사용되는 쿼리는 캐싱 기법을 활용하거나, Materialized View 등을 사용하세요.

---

## ⚙️ 3. **설정 최적화 (postgresql.conf)**

### 📦 메모리 관련 설정
- `shared_buffers`: 전체 RAM의 25% 정도로 설정 (예: 2GB)
- `effective_cache_size`: 전체 시스템 메모리의 50~75%
- `work_mem`: 각 작업에 할당할 메모리 (예: 64MB)

```conf
shared_buffers = 2GB
effective_cache_size = 6GB
work_mem = 64MB
```

### 🔄 WAL 및 백업 설정
- `wal_buffers`, `checkpoint_completion_target` 등을 조정하여 I/O 성능 개선 가능.

---

## 🗃️ 4. **테이블 구조 최적화**

### 📦 테이블 분할 (Partitioning)
- 대용량 테이블은 날짜나 범주별로 파티셔닝하여 성능 향상.

```sql
CREATE TABLE orders_2024 PARTITION OF orders FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');
```

### 🧹 정규화 vs 반정규화
- 필요에 따라 정규화 또는 반정규화를 고려하세요.
- 예: JOIN이 많아질 경우 denormalization으로 성능 개선.

---

## 🛠️ 5. **확장성 및 병렬 처리**

### 🧠 병렬 쿼리
- `max_parallel_workers_per_gather` 설정으로 병렬 처리 활성화.

```conf
max_parallel_workers_per_gather = 4
```

### 🧩 분산 처리
- PostgreSQL 12 이상에서는 `pg_partman` 같은 확장 모듈을 통해 파티셔닝과 분산 처리 가능.

---

## 📈 6. **모니터링 및 성능 분석 도구**

### 🧪 `pg_stat_statements`
- 자주 실행되는 쿼리와 성능 정보를 수집합니다.

```sql
CREATE EXTENSION IF NOT EXISTS pg_stat_statements;
SELECT * FROM pg_stat_statements ORDER BY calls DESC LIMIT 10;
```

### 📊 `pgBadger`, `pgHero`, `pgAdmin` 등
- 로그 분석 및 성능 모니터링 도구 사용.

---

## 🧹 7. **정리 및 VACUUM / ANALYZE**

### 🧹 VACUUM & ANALYZE
- 삭제된 행을 정리하고 통계 정보를 갱신해야 합니다.

```sql
VACUUM ANALYZE table_name;
```

- `autovacuum` 설정을 통해 자동으로 처리되도록 설정 가능.

---

## 🧠 8. **트랜잭션 및 잠금 관리**

- 긴 트랜잭션은 성능 저하 원인이 될 수 있습니다.
- 필요한 만큼의 트랜잭션만 사용하고, 잠금 시간을 최소화하세요.

---

## 🧾 요약

| 항목 | 설명 |
|------|------|
| 인덱스 | WHERE, JOIN, ORDER BY에 사용되는 컬럼에 인덱스 추가 |
| 쿼리 최적화 | EXPLAIN ANALYZE로 실행 계획 확인 |
| 설정 최적화 | shared_buffers, work_mem, effective_cache_size 조정 |
| 파티셔닝 | 대용량 테이블에 대해 날짜/범주별 분할 |
| 캐싱 | 자주 사용되는 쿼리 캐싱 또는 Materialized View 사용 |
| VACUUM/ANALYZE | 정기적으로 테이블 정리 및 통계 갱신 |
| 모니터링 | `pg_stat_statements` 등으로 성능 분석 |
